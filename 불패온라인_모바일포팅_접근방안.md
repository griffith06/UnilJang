# 불패온라인 모바일 포팅 접근 방안 비교

> 작성일: 2026-05-02
> 전제: 불패온라인(2011-11 런칭) 풀소스 도착 대기 중. 도착 시 모바일 버전으로 개량 진행 예정.
> 가용 자산: `ZerodinEngineMDM_DX` (PC, 퀸스블레이드급), `ZerodinEngineMDM` (모바일 검증), `원더랜드_Program` (하트캐슬 실 런칭 모바일 게임)

---

## 결론 요약

**추천: B안 (불패 엔진을 MDM_DX 기준으로 정렬 → MDM 모바일 패치 재적용)** 을 베이스로 하되,
네트워크 / UI / 패킹 / 서버 파이프라인은 A안의 원더랜드 자산을 흡수하는 **하이브리드** 가 가장 현실적.

이유는 세 가지:

1. 불패는 MMO 성격이 강해 원더랜드(단일플레이) 프레임만으로는 부족함
2. 원더랜드의 모바일 빌드/패킹/서버 파이프라인은 이미 검증되어 있어 재발명할 필요 없음
3. C안(엔진 전면 교체)은 12년치 Zerodin 노하우/툴체인을 버리게 됨

---

## A안 — 원더랜드 프레임 위에 불패 게임 코드 이식

원더랜드(`원더랜드_Program`)를 베이스로 두고, 불패의 게임 로직(스킬 / 직업 / 퀘스트 / 인벤토리 등)만 뽑아 이식.

### 장점

- 모바일 빌드 파이프라인 즉시 가동 가능 (`ZerodinMobilePacker.exe`, iOS 빌드, ClientBin/ServerBin 구조 그대로)
- PVRTexTool 등 모바일 전용 텍스처 처리 이미 셋업됨
- 검증된 모바일 메모리/배터리 프로파일 재활용

### 단점

- 불패 PC 전용 시스템(BSP 맵, 대규모 필드, MMO 채팅/파티/길드/거래 시스템) 사실상 재작성
- 불패 원본 맵 자산 활용도가 가장 낮음 (BSP → Entity 수동 컨버팅 필요)
- 불패의 "맛"이 원더랜드 프레임에 갇히게 될 위험

### 적합한 경우

- 불패의 기획/IP만 살리고 게임 구조는 처음부터 모바일에 맞게 재설계할 거라면.
- 일정이 매우 타이트할 때.

## B안 — 불패 엔진을 MDM_DX 기준으로 먼저 정렬 → MDM 모바일 패치 재적용 ★ 추천

불패(2011) 엔진은 퀸스블레이드(2012)보다 살짝 이전 버전. 따라서:

1. 불패 엔진 ↔ `ZerodinEngineMDM_DX` 의 GfEngine / GfDXUT / GfExEngine 차이를 정리
2. `ZerodinEngineMDM_DX` → `ZerodinEngineMDM` 변환 시 적용했던 패치(BSP → Entity, DX → GLES, GfCCube/GfPhysics/GfVolume/GfSpeedTree 제거 등)를 거의 그대로 재적용
3. 게임 로직 코드(불패 고유 콘텐츠)는 최대한 원본 유지

### 장점

- 불패 원본 콘텐츠/맵/리소스 재활용도 최대
- 이미 한 번 검증된 변환 패턴(MDM_DX → MDM) 재사용 → 리스크 최소
- 게임 본질(불패다움) 보존
- DX 엔진 레이어만 갈아끼우는 작업이라 게임 로직 회귀 위험 낮음

### 단점

- DX9~10 시절 셰이더 / 렌더 패스를 모바일 GLES(또는 Metal/Vulkan)로 다시 풀어야 함
- BSP 기반 맵을 Entity 기반으로 컨버팅하는 자동화 도구 필요 (또는 수작업)
- 불패-MDM_DX 엔진 버전 차이 분량에 따라 초기 정렬 작업 공수가 큼

### 권장 단계

1. 불패 소스 도착 직후: `Engine/Core/*` 디렉토리 구조 비교, 모듈별 diff
2. 게임 로직 ↔ 엔진 레이어 경계 식별 (게임 코드는 손대지 않을 영역 확정)
3. 엔진 레이어를 MDM_DX 버전으로 머지
4. MDM 모바일 패치 적용 (`Engine/Core` 슬림화, GLES 렌더러 교체)
5. 원더랜드의 MobilePacker / iOS 빌드 / 서버 파이프라인 흡수 (하이브리드 부분)

## C안 — 자산만 추출하고 엔진은 현대(Unity / Unreal)로 재작성

불패의 모델 / 텍스처 / 사운드 / 기획 데이터만 추출하고, 엔진 자체는 Unity 또는 Unreal로 새로 만듬.

### 장점

- 2026년 iOS/Android 스토어 정책에 깔끔하게 부합 (64bit, Metal/Vulkan, 최신 SDK)
- 멀티플랫폼(스위치/PC/콘솔 동시) 옵션 열림
- 신규 인력 합류 시 학습 곡선 낮음 (Unity/Unreal 인력 풀이 큼)
- 셰이더 / 라이팅 / UI 시스템 등 현대적 기능 무료로 따라옴

### 단점

- 12년치 Zerodin 노하우 / 자체 툴체인 / 컨버터 거의 사용 안 됨
- 사실상 신규 프로젝트와 동급의 공수
- 불패의 클라이언트-서버 프로토콜을 Unity 클라에 다시 구현해야 함
- "Zerodin으로 만든 불패의 모바일 버전"이 아니라 "불패 IP를 쓴 신작" 에 가까워짐

### 적합한 경우

- 어차피 IP만 살리고 모든 걸 새로 만들 예정이라면.
- 장기적으로 멀티플랫폼/콘솔까지 노린다면.

## 소스 도착 전 미리 할 수 있는 일

A/B/C 중 어떤 길을 가더라도 도움이 되는 사전 작업:

1. **MDM_DX ↔ MDM 변환 매핑 문서화**
   어떤 모듈을 빼고, 어떤 함수를 GLES로 치환했는지 정리.
   (B안 추진 시 그대로 재사용, A/C안 추진 시에도 무엇을 모바일에서 떨궜는지 참고용)

2. **기존 문서 정리**
   `원더랜드_Program/MDM모바일패킹및다운로드시스템에관련해서.docx`,
   `메릴린 서버클라 주요 클래스다이어그램.uml`,
   `메릴린 월드 클래스다이어그램.uml`,
   `ZerodinEngineMDM_DX/기타등등/바꾸는것들.txt`,
   `ZerodinEngineMDM_DX/기타등등/쉐도우 관련브리핑.docx`
   → 통합된 "포팅 체크리스트" 한 장으로

3. **2026년 타겟 정책 결정**
   
   - iOS / Android 최소 지원 OS 버전
   - 64bit only 강제 대응
   - Metal / Vulkan 대응 여부 (구 GLES2 only 빌드는 스토어 정책상 막히는 부분 발생 가능)
   - 텍스처 압축 포맷 갱신 (PVRTC는 deprecated, ASTC 권장)

4. **불패 원본 미리보기**
   소스 도착 전이라도 불패 클라이언트 리소스(맵 / 모델) 샘플을 받을 수 있다면,
   BSP 기반 맵의 모바일 변환 가능성(폴리곤/드로우콜 예산)을 미리 측정.
