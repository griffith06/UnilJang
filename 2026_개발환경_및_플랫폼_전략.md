## 1. 그래픽스 API 전략 (가장 먼저 결정해야 할 것)

### 1.1 2026년 시점 선택지

| 플랫폼          | 2026년 권장               | 비고                                                          |
| ------------ | --------------------- | ----------------------------------------------------------- |
| Windows PC   | **DX11 확정**           | DX9는 단종. DX11이 안정적이고 GfDXUT 자산 재활용 가능. DX12는 사실상 재작성 수준이라 3인팀 ROI 낮음 |
| iOS          | **Metal 필수**          | OpenGL ES는 iOS 12부터 deprecated, App Store 신규 GLES 등록 제한    |
| Android      | **GLES3 단일 확정**       | Android 4.3+ 99% 커버, GfOpenGlES 자산 100% 재활용, 중국 디바이스 호환성 우수. **Vulkan은 런칭 후 추가 작업 여부 별도 결정** |
| macOS        | **(선택) Metal**        | PC 빌드 시 보너스로 가능                                             |

### 1.2 추천 안 (3인팀 기준) ✅ **확정**

**DX11 (PC) + Metal (iOS) + GLES3 (Android)** — 백엔드 3개

#### DX11 채택 사유 (DX12 대비)

DX9 → DX11과 DX9 → DX12는 마이그레이션 공수가 차원이 다름:

| 항목             | DX9 → DX11        | DX9 → DX12                     |
| -------------- | ----------------- | ------------------------------ |
| 패러다임           | 점진적 진화            | **완전 재작성** (low-level/explicit) |
| 자원 관리          | 드라이버 자동           | 수동 (Descriptor Heap, Fence 등)  |
| 마이그레이션 공수      | 중간                | **5~10배**                       |
| MMO 실익         | 충분                | 거의 없음 (CPU 바운드)                |
| GfDXUT 자산 재활용  | 가능                | 사실상 불가                          |

→ MMO는 GPU 드로우콜 한계 안 침. DX12 도입은 3인팀 공수만 잡아먹고 실익 없음.
→ DX12는 추후 콘솔 확장 검토 시 재논의.

#### GLES3 단일 채택 사유 (Vulkan 미채택)

| 항목                      | GLES3 단일                       | Vulkan + GLES3 폴백        |
| ----------------------- | ----------------------------- | ------------------------ |
| 백엔드 개수                  | 3개 (전체)                       | 4개 (전체)                  |
| 셰이더 파이프라인               | GLSL 한벌                       | GLSL + SPIR-V 두 벌       |
| GfOpenGlES 자산 재활용도      | **100%** (원더랜드 검증)            | 부분                       |
| 성능 (방치형 MMO 기준)         | 60fps 충분                      | 30~50% 더 빠르나 cap 무의미 |
| 신형 GPU 드라이버 안정성         | 검증 끝, 버그 거의 없음                | 일부 미들급에서 가끔 보고          |
| Android 디바이스 커버리지       | 99%+ (Android 4.3+)           | Vulkan 부분만 7+            |
| 3인팀 공수 절감 효과            | RHI/셰이더/디버깅/QA 누적 큼          | -                        |

채택 사유 요약:
- 방치형 MMO는 GPU 바운드 아님 → Vulkan 성능 이점 무의미
- GfOpenGlES 자산이 가장 큰 무기. "재컴파일 + 셰이더 SM5 업그레이드" 수준으로 안드로이드 백엔드 가능
- 한국 시장은 사실상 삼성 단일 OEM이라 GLES 드라이버 안정성 매우 높음
- Google은 아직 GLES를 명시적으로 deprecate하지 않음. 5년은 안전
- 3인팀 공수 절감 효과 누적 큼 (RHI + 셰이더 + 디버깅 + QA)

#### **Vulkan 도입 시점: 런칭 후 별도 결정**

Vulkan은 폐기가 아닌 **2단계 로드맵**으로 처리:
- **Phase 1 (런칭)**: GLES3 단일. 빠른 출시, 호환성 안전망 우선.
- **Phase 2 (런칭 후 6개월~1년)**: 시장 반응 / 신형 디바이스 비중 / GLES3 deprecation 동향을 보고 Vulkan 백엔드 추가 작업 여부 별도 결정. 추가 시 옵션 토글 형태로 점진 적용.

판단 기준 (Phase 2 진입 시 검토):
- Google이 GLES deprecate 공식화 했는가?
- 120fps 지원 요구가 시장에서 강해졌는가?
- 신형 갤럭시(Xclipse 등)의 GLES 드라이버 업데이트가 끊겼는가?
- 글로벌 진출 결정 시 해당 시장의 GPU 분포가 GLES만으로 부족한가?

- Metal은 iOS는 선택지가 없음 (불변)

### 1.3 통합 렌더링 추상화 레이어 (RHI) 결정

GLES3 단일 채택으로 백엔드가 3개로 줄어 RHI 부담 감소:

| 옵션                         | 장점                  | 단점                            |
| -------------------------- | ------------------- | ----------------------------- |
| **자체 RHI 작성** (백엔드별 구현)    | 완전 통제, 엔진 일체감, 자산 최대 활용 | 3개 백엔드 작성 (Vulkan 빠져 부담 줄음)  |
| **bgfx 등 OSS 도입**          | 검증된 다중 백엔드          | 자체엔진 철학과 충돌, 디버깅 복잡          |
| **셰이더 크로스 컴파일만 도입**       | HLSL 한벌 작성 → GLSL/MSL 변환 | RHI는 자체 작성, 셰이더 부담만 줄임 ★ 추천 |

**추천: 셰이더는 HLSL 한벌로 작성하고 SPIRV-Cross로 GLSL/MSL 변환. RHI는 자체 작성하되 GfDXUT/GfOpenGlES 자산 최대 재활용.**

→ Vulkan 미채택으로 SPIR-V 중간 단계 부담은 줄지만, MSL/GLSL 변환에는 여전히 SPIRV-Cross 경로가 표준. 셰이더 한벌 정책은 유지.

## 2. Android 파편화 대응 (한국 시장 기준)

> **참고**: 2012~2016년 모바일 시기에 중국 디바이스 호환에 큰 노력을 들였던 경험이 있으나, **이번 프로젝트는 한국 시장 우선이라 중국 디바이스는 타겟에서 제외**. 글로벌 진출은 런칭 후 별도 검토 사항.

### 2.1 2012~2016 vs 2026 비교 (한국 안드로이드 시장 기준)

| 항목                    | 2012~2016                                       | 2026                                                |
| --------------------- | ----------------------------------------------- | --------------------------------------------------- |
| OS 파편화                | Android 4.0~6.0 공존, 극심                          | 13~16 중심, **Play 정책상 targetSdk 강제**로 안정          |
| 한국 OEM 다양성            | 삼성, LG, 팬택, 베가, KT 테크, ...                      | **사실상 삼성 단일** (LG 단종, 픽셀/모토로라 점유율 미미)             |
| GPU 종류                | Adreno/Mali/PowerVR/Tegra/Vivante 등 5종+         | **Adreno (구글픽셀/삼성 일부) + Mali (삼성 미들) + Xclipse (삼성 플래그십)** 3종 수렴 |
| 해상도 파편화               | 480p~1080p 다양                                   | 1080p~QHD+ 표준화. 폴더블/펀치홀 노치 추가 (Galaxy Z 시리즈)     |
| 텍스처 포맷                | ETC1 / PVRTC 분리                                 | **ASTC 단일** (모든 모던 GPU 지원)                        |

| OEM 커스텀 ROM           | 제조사별 매우 다름                                      | One UI(삼성) 사실상 단일 + AOSP 픽셀                       |

- 중국 본토: 본 프로젝트 타겟 외

- XTP1500 (Codejock) MFC 툴 → 최신 버전 라이선스 확인 필요
- DXUT는 deprecated, GfDXUT 자체 코드만 유지

## 4. 자체엔진 현대화 체크리스트

기존 ZerodinEngineMDM_DX 기준에서 손봐야 할 것:

### 4.1 기반 시스템
- [ ] 32bit → **64bit 전환** (포인터/사이즈 타입 전수 점검)
- [ ] 모던 C++ 마이그레이션 (C++03 잔재 → C++17)
  - `auto`, range-for, `nullptr`, smart pointer, `std::filesystem` 등
- [ ] 멀티스레드 렌더링 (RenderThread 분리)
- [ ] Job System / Task Graph 도입 (엔티티/AI/물리 병렬화)

## 5. MMO 모바일 게임플레이 고려사항

PC MMO를 모바일로 옮길 때 단순 이식이 아니라 **게임 자체를 재설계**해야 하는 영역.

### 5.1 자동사냥 (모바일 MMO 필수) ✅ **방치형 확정**

타겟층(30~50대, 시간 없음)에 맞춰 **풀오토(방치)** 컨셉 확정.

| 항목                  | 설계 방향                                                                                      |
| ------------------- | ------------------------------------------------------------------------------------------ |
| 자동사냥 강도             | **풀오토** - 사냥터 지정만 하면 자동으로 돈다                                                              |
| 사냥터 자동 이동           | NavMesh 기반 길찾기. 죽으면 자동 부활 후 사냥터 복귀                                                       |
| 자동 스킬 우선순위          | 유저 설정 UI (스킬 슬롯 끌어다 우선순위, 쿨다운 자동)                                                         |
| 자동 회복 / 포션          | 체력/마나 임계치 설정, 자동 사용. 떨어지면 마을 귀환                                                          |
| 자동 인벤토리 관리          | 등급별 자동 판매 / 분해 / 보관 규칙 설정                                                                |
| 자동 사냥 중 PvP         | 방어 / 도망 / 반격 / 안전지대 귀환 모드 선택                                                              |
| 백그라운드 자동사냥          | ✅ **서버측 시뮬레이션 확정** - 클라가 꺼져도 캐릭터는 서버에서 계속 사냥. iOS/Android 백그라운드 제약 무관    |
| 보상 누적 박스            | 4/8/12시간 방치 보상 (오프라인 보상)                                                                |
| 사망 / 위급 푸시 알림       | 보스/PK 사망, 인벤 가득참, 회복약 부족 시 푸시                                                            |
| 어뷰징 방지              | 다중 계정 방치 농장 탐지, IP/디바이스 핑거프린팅                                                            |

**서버측 시뮬레이션 확정 - 파급 효과**:

이 결정으로 서버 아키텍처가 일반 MMO와 완전히 달라짐. 자세한 내용은 신설된 [9장. 서버 아키텍처](#9-서버-아키텍처-방치형-mmo-구조) 참고.

핵심 효과:
- ✅ 진짜 방치형 컨셉 완성 ("PC 켜놓고 출근")
- ✅ 어뷰징 봇 가치 소멸 (모두 자동사냥이라 봇과 구분 무의미)
- ⚠️ 서버 비용 일반 MMO 대비 5~10배
- ⚠️ 사냥터 24시간 경쟁 → 인스턴싱 / 채널링 필수
- ⚠️ ECS 또는 Actor 모델 기반 서버 재설계 필요 (안명달 이사 영역)

### 5.1.1 타겟층(30~50대 아저씨) 특화 UX
- **큰 글씨 / 큰 버튼** 옵션 (시력 배려)
- **한 손 조작 모드** (출퇴근 지하철)
- **PC 모드 = 멀티 캐릭터 자동사냥 / 다창** 허용 검토 (대다수가 PC 켜놓고 출근하는 패턴)
- **간소화된 빌드/스킬** - 복잡한 트리보다 직관적 분기
- **길드/사회 컨텐츠 강조** - 옛 동료 / 향수 마케팅
- **결제 UI 단순화** - 부담 없이 한 번에 결정 가능한 패키지 위주
- **자동사냥 영상 보내기** - 카톡/문자로 "내 캐릭 이렇게 강해졌어" 자랑 기능

→ **불패 PC 버전에는 거의 없는 시스템. 신규 설계 필요.** 안명달 이사가 서버측 시뮬레이션 가능 여부, 최동순이 UX/밸런스 영역 책임.

### 5.2 성장 곡선 재설계
- PC MMO: 한 자리에서 1~2시간 사냥, 레벨업 만족
- 모바일 MMO: 5~10분 짧은 세션 + 자동사냥 누적
- **경험치 곡선, 보상 빈도, 일일 컨텐츠 양 전체 재밸런싱**

### 5.3 PC + 모바일 크로스플레이 vs 분리

| 옵션                                | 사례                                | 장점                  | 단점                            |
| --------------------------------- | --------------------------------- | ------------------- | ----------------------------- |
| **① 완전 통합** (같은 서버 / 같은 캐릭터 / 같은 필드) | 원신, 미르4, 오딘, 디아블로 이모탈           | 듀얼 런칭 시너지 최대, 마케팅 통합, "PC 방치 → 모바일 외출 시 이어하기" 자연스러움 | PC/모바일 밸런스, 컨트롤 차이 보정, 어뷰징 |
| **② 계정만 공유 / 서버 분리**              | 거의 안 쓰임                          | 운영 유연              | 친구와 못 만남, 어정쩡함                |
| **③ 완전 분리** (IP만 공유)              | 리니지 vs 리니지M, 메이플 vs 메이플M           | 한쪽씩 독립 설계 가능       | 듀얼 런칭이 두 게임 만드는 꼴 → 3인팀 무리 |

**✅ 옵션 ① 완전 통합 확정**:
- "출근 시 PC 자동사냥 → 점심/퇴근 모바일로 보스/이벤트 챙기기" 시나리오가 핵심 가치제안
- 같은 캐릭터/세계라 두 클라이언트 간 콘텐츠 차이만 관리하면 됨 → 두 게임을 따로 설계하는 것보다 공수 적음
- 서버측 자동사냥 시뮬레이션(5.1)과 자연스럽게 결합

**옵션 ① 채택 시 풀어야 할 숙제 - 결정 완료**:
- ✅ **PC 자동사냥 허용** (모바일과 동일 시스템)
- ✅ **PC 다중 클라이언트 차단** (1계정 1클라 강제)
- 컨트롤 보정: PC/모바일 모두 자동사냥이라 차이 작음. 수동 조작 시 차이만 보정
- 채팅/길드 UI는 양쪽이 다른데 메시지는 공통

### 5.4 컨트롤 / UI
- 가상 조이스틱 vs 타겟 클릭 (한국 MMO는 후자 다수)
- 스킬 슬롯 개수 (모바일 화면 한계: 6~8개가 표준)
- 채팅 UX (모바일 키보드 가림 처리)
- 미니맵 / 인벤토리 / 상점 UI 전체 재설계

### 5.5 과금 모델
- 정액 → 부분유료 (확률형 + 패스 + 재화)
- 확률 공시 의무 (한국 게임산업법, 2024년 시행)
- 청소년 보호 / 셧다운제 폐지 후 시간/결제 한도 등 점검

### 5.6 운영 시스템
- 라이브 운영 도구 (이벤트, 점검, 핫픽스)
- 어뷰징/매크로 탐지 (모바일은 PC보다 더 심함)
- CS 인입 채널

## 9. 서버 아키텍처 (방치형 MMO 구조)

5번 결정(서버측 자동사냥 시뮬레이션)으로 신설된 장. 안명달 이사 주도 영역.

### 9.1 일반 MMO vs 방치형 MMO 비교

| 항목              | 일반 MMO                         | 방치형 (불패M)                       |
| --------------- | ------------------------------ | ------------------------------- |
| 활성 캐릭터 수        | 동시접속자 = 활성                     | **누적 가입자 ≈ 활성** (다 24시간 사냥)    |
| 서버 비용           | 동접 기준                          | 동접 기준 **5~10배**                |
| DB 갱신           | 이벤트 기반                         | 메모리 우선 + 주기적 스냅샷               |
| 사냥터 경쟁          | 동접 시간대만 격함                     | **24시간 격함** → 인스턴싱 필수         |
| 봇 vs 정상 유저      | 패턴으로 구분 가능                     | 사실상 구분 불가 → 어뷰징 정책 재정의       |
| 서버 아키텍처         | 단일 프로세스/존 단위 가능               | **ECS 또는 Actor 모델 사실상 강제**    |

### 9.2 서버 아키텍처 패턴 (논의 필요)

| 옵션                                | 장점                              | 단점                                |
| --------------------------------- | ------------------------------- | --------------------------------- |
| **ECS (Entity Component System)** | 캐시 친화, 대량 엔티티 시뮬에 최적, 확장 쉬움    | 기존 OOP 서버 코드와 패러다임 충돌, 학습 곡선  |
| **Actor 모델 (Akka.NET, Orleans)**  | 분산 자연스러움, 격리 안전               | 메시지 패싱 오버헤드, 핫스팟 처리 어려움         |
| **하이브리드** (사냥터=ECS / 채팅·거래=Actor) | 영역별 최적화                        | 두 패러다임 다뤄야 함, 경계 설계 복잡          |

→ 안명달 이사 검토 필요. 불패 PC 서버가 어떤 구조였는지 따라 결정 갈림.

### 9.3 사냥터 인스턴싱 / 채널링

24시간 자동사냥하면 사냥터가 항상 포화. 옵션:
- **무한 채널링**: 인구 임계 넘으면 자동으로 채널 추가/삭제. 같은 채널 친구 못 만남 단점.
- **사냥터 슬롯제**: 사냥터당 자리 N개, 풀이면 대기열. 자리 경쟁 자체가 컨텐츠 (PvP 발생).
- **개인 인스턴스 던전**: 자동사냥 전용 던전은 1인/파티 인스턴스, 필드 사냥은 채널링.
- **레벨대 분리**: 사냥터를 레벨 단위로 잘게 나눠 자연 분산.

→ 게임 디자인(최동순 영역)과 직결.

### 9.4 DB 정책

24시간 시뮬되는 캐릭터 N명 → DB가 매 틱마다 갱신되면 폭발. 권장:
- **메모리 마스터**: 캐릭터 상태는 메모리에 두고, **N분 단위 스냅샷**만 DB에 기록
- **이벤트 기반 즉시 기록**: 결제, 거래, 보스 드랍 등 손실 허용 안되는 것만
- **정전 시 손실 허용 범위**: 사냥 5분치 손실 허용? 1분치만? - 비용/안정성 트레이드오프

### 9.5 어뷰징 / 봇 정책 재정의

기존 MMO: 자동사냥 = 봇 = 차단.
방치형: 자동사냥 = 정상.

새로 정의할 어뷰징:
