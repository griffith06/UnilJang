# x64 전환 — pack(1) 제거 및 정공법 수정 작업 명세서

> **목표**: vcxproj의 x64 `StructMemberAlignment: 1Byte` (= 전역 pack(1))를 제거하고
> default pack(8)로 전환. pack(1)이 반드시 필요한 구조체만 개별 `#pragma pack(push,1)` / `#pragma pack(pop)`으로 보호한다.

## 작업 분류 개요

pack(1)을 제거하면 구조체에 **패딩(padding)**이 삽입된다.
영향 받는 구조체는 크게 **4가지 카테고리**로 나뉜다:

| 카테고리 | 설명 | 조치 |
|----------|------|------|
| **A. 네트워크 패킷** | 서버와 `send()/recv()`로 주고받는 구조체. sizeof가 서버와 일치해야 함 | **반드시 pack(1)** |
| **B. 파일 I/O** | `fread()/fwrite()`로 구조체 통째 읽기/쓰기. 기존 파일 호환 필요 | **반드시 pack(1)** |
| **C. LPVOID 포인터** | 구조체 내 LPVOID/포인터 멤버. x86=4바이트, x64=8바이트로 크기 변동 | **LPVOID → OLD_LPVOID (= DWORD) 교체** (별도 섹션) |
| **D. 내부 전용** | 메모리에서만 사용, 네트워크/파일 I/O 안 함 | **pack 불필요** (default OK) |

## A. 네트워크 패킷 구조체 — pack(1) 필수

> 서버(32비트)와 바이트 단위 호환 필수. `SendMapServer((char*)&msg, sizeof(sMSG_*))` 패턴으로 사용.
> **반드시 `#pragma pack(push,1)` / `#pragma pack(pop)`으로 감싸야 한다.**

### A-1. LPVOID 포함 패킷 (가장 위험 — C 카테고리와 교차)

이 구조체들은 pack(1)뿐만 아니라 **LPVOID를 OLD_LPVOID (= DWORD, 4바이트 고정)로 교체**해야 한다.
서버는 32비트이므로 LPVOID 자리에 4바이트를 보낸다. x64 클라이언트에서 LPVOID는 8바이트가 되어 **패킷 파싱이 완전히 깨진다.**

**조치:**

먼저 한 곳에 `OLD_LPVOID`를 선언한다 (나중에 어디를 고쳤는지 추적 가능):
```cpp
// Common.h 상단 또는 별도 공통 헤더
// x64 전환: 기존 LPVOID(서버 32비트 식별자)를 4바이트 고정 타입으로 교체
// 검색 키워드 OLD_LPVOID로 변경된 위치를 모두 추적 가능
typedef DWORD OLD_LPVOID;
```

```cpp
// BEFORE (x86에서 LPVOID = 4바이트, x64에서 8바이트 → 서버와 불일치)
typedef struct{
   BYTE     tag;
   LPVOID   who;        // ← 문제
   int      status;
}sMSG_PLAYER_STATUS;

// AFTER (OLD_LPVOID = DWORD = 항상 4바이트, 서버와 일치)
#pragma pack(push, 1)
typedef struct{
   BYTE        tag;
   OLD_LPVOID  who;        // ← 서버에서 보내는 4바이트 식별자로 고정
   int         status;
}sMSG_PLAYER_STATUS;
#pragma pack(pop)
```

> **핵심**: 서버가 보내는 `who`는 **서버 측 포인터 값(4바이트)**이며, 클라이언트에서는 이를 식별자(핸들)로만 사용한다.
> 따라서 `LPVOID` → `OLD_LPVOID` (= DWORD)로 교체해도 로직 문제 없다.
> `OLD_LPVOID`로 grep하면 x64 전환으로 변경된 모든 위치를 즉시 확인할 수 있다.
> 단, 클라이언트 코드에서 이 값을 실제 포인터로 캐스팅해서 사용하는 부분은 **별도 매핑 테이블** 방식으로 전환해야 한다. (C 카테고리 참조)

## B. 파일 I/O 구조체 — pack(1) 필수

> `fread(&struct, sizeof(struct), 1, fp)` 패턴으로 구조체 통째 읽기/쓰기하는 경우.
> 기존 파일과의 바이너리 호환성을 위해 pack(1) 유지 필수.

### B-1. sFileBind (Char.Off 인덱스)

```cpp
// Sprite.h:35, Mob.h:10
typedef struct{
   char fileName[21];   // 21 bytes
   int  fileOffset;     // 4 bytes
   int  fileSize;       // 4 bytes
}sFileBind;              // pack(1) 시 29 bytes, default 시 32 bytes (패딩 3바이트)
```

**사용처:**
- `Sprite.cpp:95` — `fin.read((char*)sPCHAR_OFF, sizeof(sFileBind) * gCHAR_OFF_CNT)`
- `Sprite.cpp:110` — `fread(sPCHAR_OFF, sizeof(sFileBind), gCHAR_OFF_CNT, fp)`
- `RESTools/MapDoc.cpp:1519` — `fread(m_bindOffsets, sizeof(sFileBind), m_nChaDataCount, fp)`

**조치:** `#pragma pack(push,1)` / `#pragma pack(pop)`으로 감싸기.
- Char.Off 파일은 29바이트 × 22,487 = 637KB로 기록되어 있으므로 반드시 29바이트 유지

### B-2. DDS 헤더 (StaticTexture.cpp)

```cpp
// StaticTexture.cpp:24-60 — 이미 pack(push,1) 적용됨 ✅
#pragma pack(push, 1)
struct FDDSHeader { ... };    // 124 bytes 고정
struct FDDSHeaderDXT10 { ... }; // 20 bytes 고정
#pragma pack(pop)
```

**상태:** 이미 올바르게 처리됨. 수정 불필요.

### B-3. PE 구조체 (Dll.h)

```cpp
// Dll.h:60-85 — 이미 pack(push,1) 적용됨 ✅
#pragma pack(push,1)
typedef struct { DWORD dwPageRVA; DWORD dwBlockSize; } IMAGE_FIXUP_BLOCK;
typedef struct { WORD offset:12; WORD type:4; } IMAGE_FIXUP_ENTRY;
typedef struct { void *pImageBase; char svName[MAX_PATH]; DWORD dwFlags; } IMAGE_PARAMETERS;
#pragma pack(pop)
```

**상태:** 이미 올바르게 처리됨. 단, `IMAGE_PARAMETERS`에 `void*` 포함 — x64에서 크기 변동. CDLL이 미사용이면 제거 권장.

### B-4. ICMP/IP 헤더 (Ping.cpp)

```cpp
// Ping.cpp:13 — #pragma pack(4) 적용됨 ✅
// IpHeader, _ihdr 구조체는 네트워크 프로토콜 표준이므로 pack(4) 유지
```

**상태:** 유지. 수정 불필요.

### B-5. FGP 파일 — sizeof(LPBYTE) 치명적 버그

```cpp
// RESTools/FieldSprite.cpp:296
fread(&location, sizeof(LPBYTE), 1, fp);   // x86=4바이트, x64=8바이트 → 깨짐!

// RESTools/ObjMainView.cpp:2104
fwrite(&location, sizeof(LPBYTE), 1, fp);

// RESTools/ObjMainView.cpp:2148
fseek(fp, (4 + sizeof(WORD) + sizeof(WORD) + sizeof(LPBYTE)), SEEK_SET);

// RESTools/Sprite.cpp:1889
fread(&location, sizeof(LPBYTE), 1, fp);
```

**조치:**
- `sizeof(LPBYTE)` → `sizeof(uint32_t)` 또는 `4` (고정 크기)로 교체
- RESTools는 Win32 유지이므로 당장은 문제없지만, 코드 안전성 위해 수정 권장
- FGP v2 포맷 도입 시 반드시 수정

### B-6. SPR 파일 — 필드별 fread (현재 안전)

```cpp
// Sprite.cpp — 필드별 개별 읽기라 pack 영향 없음 ✅
```

## C. LPVOID 포인터 문제 — 크기 변동 (x86=4, x64=8)

> LPVOID(= `void*`)는 x86에서 4바이트, x64에서 8바이트.
> 구조체 크기가 변하면 네트워크 패킷 파싱, 메모리 레이아웃이 모두 깨진다.

### C-1. 네트워크 패킷 내 LPVOID (A-1과 교차 — 가장 위험)

**위 A-1 섹션의 모든 구조체가 해당.**

서버(32비트)는 LPVOID 자리에 4바이트 값을 보낸다.
클라이언트(64비트)에서 이를 받을 때 LPVOID가 8바이트이면 **패킷이 어긋난다.**

**해결 방안:**

```
방법 1 (권장): LPVOID → DWORD 교체 + 해시맵 매핑
─────────────────────────────────────────────────
서버에서 받은 4바이트 who 값 → DWORD로 저장
클라이언트에서 실제 오브젝트 참조 시: std::unordered_map<DWORD, sPLAYER*> 등으로 매핑

장점: 패킷 구조 변경 없음, 서버 수정 불필요
단점: 룩업 오버헤드 (미미)
```

```
방법 2: 래퍼 타입 정의
─────────────────────
typedef DWORD WHO_ID;  // 항상 4바이트, 서버 식별자

패킷 구조체에서:
LPVOID who;  →  WHO_ID who;

코드에서 사용 시:
sPLAYER* pPlayer = FindPlayerByWhoID(msg.who);
```

### C-2. 게임 오브젝트 구조체 내 LPVOID (내부 전용)

네트워크로 전송하지 않고 **클라이언트 메모리에서만 사용**하는 LPVOID.
이들은 pack(1) 필요 없고, x64에서 8바이트 포인터로 자연스럽게 동작한다.
단, **네트워크 패킷으로부터 값을 복사해 올 때** 타입 변환에 주의.

| 구조체 | 파일 | LPVOID 멤버 | 용도 |
|--------|------|-------------|------|
| `sPLAYER` | Player.h:86 | `client` | 클라이언트 소켓 포인터 (내부 참조) |
| `sME_ETC_PARA` | Player.h:33-34 | `mainCharP`, `client` | 자기 캐릭터 포인터 (내부 참조) |
| `sMONSTERS` | Mons.h:19 | `who` | 서버에서 받은 who ID → 로컬 참조용 |
| `sMAP_NPC` | Npc.h:33 | `who` | 서버 who ID → 로컬 참조용 |
| `sMAP_TRAP` | Npc.h:63 | `who` | 서버 who ID → 로컬 참조용 |
| `sMAP_ITEM` | Item.h:14 | `who` | 서버 who ID → 로컬 참조용 |
| `sUSERBUILDS` | UserBuild.h:14 | `who` | 서버 who ID → 로컬 참조용 |
| `sTHIEF_SMOKE` | Effect.h:44 | `who` | 이펙트 대상 참조 |
| `sMAGIC_EFF` | Magic.h:67,70 | `tagetWho`, `destWho` | 마법 대상 참조 |

| `sARENA_TEAM` | ArenaSystem.h:67 | `pPartyPtr` | 아레나 파티 참조 |

**조치:**
- 이 구조체들은 **네트워크 전송용이 아니면** pack 불필요
- 단, `sMONSTERS.who`처럼 서버에서 받은 who 값을 저장하는 경우:
  - 네트워크 수신 시 `DWORD` → `LPVOID` 캐스팅으로 대입하는 코드가 있을 것
  - x64에서는 4바이트 DWORD를 8바이트 LPVOID에 대입하는 것은 안전 (제로 확장)
  - 하지만 정확한 타입 관리를 위해 `WHO_ID` (DWORD typedef)로 통일 권장

### C-3. LPVOID 사용 코드 패턴 검색/수정 목록

```
검색해야 할 패턴:
1. (LPVOID)   — LPVOID로 캐스팅
2. ->who      — who 멤버 접근
3. ->client   — client 멤버 접근
4. .who =     — who 멤버 대입
5. == (LPVOID) — LPVOID 비교
```

## D. 내부 전용 구조체 — pack 불필요

> 네트워크/파일 I/O에 사용하지 않는 순수 메모리 구조체.
> default pack(8)로 전환해도 **기능에 영향 없다.** 오히려 정렬이 좋아져 성능 향상.

## 실행 계획

### Phase 1: Common.h 패킷 영역 pack(1) 보호 (최우선)

```cpp
// Common.h — 패킷 정의 시작 (line ~1602)
#pragma pack(push, 1)

// ... 모든 sMSG_*, sOK_MSG, sSEND_ID 등 패킷 구조체 ...

#pragma pack(pop)
// 패킷 정의 끝
```

**주의:** Common.h에는 패킷이 아닌 구조체(내부 전용)도 섞여 있다.
- `sMONSTER`, `sPARTY_INFO` 등은 LPVOID 포함 + 내부 전용이지만, 서버에서 받은 데이터를 `memcpy`로 대입하는 코드가 있을 수 있으므로 **우선 pack(1)로 포함**시키고, 이후 분리.

**작업:**
1. Common.h의 `//===== MSG =====` 블록 (line 1601~) 전체를 `pack(push,1)` / `pack(pop)`으로 감싸기
2. Common.h의 데이터 구조체 (line 1~1600 부근) 중 서버 통신에 사용되는 것도 pack(1)로 감싸기
3. `ClientSock.h`의 `sMSG_CONNECT`도 pack(1) 추가
4. `ArenaSystem.h`, `AllienceGuild.h` 등 외부 패킷 헤더도 pack(1) 추가

### Phase 2: 파일 I/O 구조체 pack(1) 보호

1. `sFileBind` (Sprite.h, Mob.h) — pack(push,1) / pack(pop) 추가
2. FGP 관련 sizeof(LPBYTE) → sizeof(uint32_t) 수정 (FieldSprite.cpp, ObjMainView.cpp, Sprite.cpp)

### Phase 3: LPVOID → DWORD(WHO_ID) 교체

1. `typedef DWORD WHO_ID;` 정의 (Common.h 상단)
2. **패킷 구조체** 내 `LPVOID who` → `WHO_ID who` 일괄 교체
3. **게임 오브젝트 구조체** 내 `LPVOID who` → `WHO_ID who` 교체
4. 코드 전체에서 who 관련 캐스팅/비교 수정:
   - `(LPVOID)pPlayer` → `pPlayer->whoID` (또는 매핑 테이블)
   - `msg.who == (LPVOID)xxx` → `msg.who == xxx`
5. `std::unordered_map<WHO_ID, sPLAYER*>` 등 매핑 테이블 구축 (필요 시)

### Phase 4: vcxproj StructMemberAlignment 제거

**이 단계는 Phase 1~3이 모두 완료된 후 실행!**

1. `GcX.vcxproj` — 4개 구성 모두에서 `<StructMemberAlignment>1Byte</StructMemberAlignment>` 제거
2. `EF.vcxproj` — 2개 구성 모두에서 제거
3. 전체 빌드 → 에러/경고 확인
4. sizeof 값 검증 (로그로 주요 구조체 크기 출력하여 서버와 일치 확인)

### Phase 5: 빌드 검증

1. Debug|x64, Release|x64 빌드 성공 확인
2. 주요 패킷 sizeof 로그 출력 → 서버(32비트 pack(1)) 값과 바이트 단위 일치 확인
3. 서버 접속 → 로그인 → 게임 진입 → 캐릭터 이동/전투 테스트
4. 기존 Win32 빌드도 정상 확인

## 위험도 평가

| 항목 | 위험도 | 이유 |
|------|--------|------|
| LPVOID 포함 패킷 수정 | 🔴 **높음** | 서버와 바이트 단위 일치 필수, 한 곳이라도 누락 시 크래시 |
| pack(1) → 개별 pragma 전환 | 🟡 **중간** | 누락 시 sizeof 불일치로 패킷 파싱 오류 |

| 내부 전용 구조체 | 🟢 **낮음** | pack 변경해도 기능 영향 없음 |

## 참고: 서버 측 영향

- 서버가 **32비트**인 경우, 서버 측 LPVOID는 항상 4바이트
- 클라이언트만 x64로 전환하면 **클라이언트 측만 수정**하면 됨
- 서버를 x64로 전환할 경우, 서버 측에도 동일한 LPVOID → DWORD 작업 필요
- 현재 작업은 **클라이언트(GcX.exe) x64 전환만** 대상
